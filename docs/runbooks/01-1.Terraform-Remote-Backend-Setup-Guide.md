# Terraform Remote Backend Setup Guide (S3 + DynamoDB)

## Objective

Configure Terraform so that:

* Local machine and Jenkins use the same Terraform state
* State is stored centrally in AWS
* Multiple users/processes cannot modify state simultaneously
* Infrastructure deployments are consistent and reliable

---

# Why We Need a Remote Backend

## Problem with Local State

By default Terraform stores state locally:

```text
terraform.tfstate
```

If state remains on a developer laptop:

* Jenkins cannot see infrastructure changes
* Team members cannot share state
* State can be lost if a machine fails
* Infrastructure drift occurs
* Concurrent deployments become dangerous

Example:

Developer creates an EKS cluster locally.

Jenkins later runs:

```bash
terraform apply
```

Jenkins has no knowledge of the existing infrastructure because its state file is different.

This can result in:

* Duplicate resources
* Failed deployments
* Resource destruction
* State corruption

---

# Solution

Use:

1. Amazon S3
2. Amazon DynamoDB

Together they provide:

* Shared state storage
* State locking
* Version history
* High durability

---

# Step 1 - Create S3 Bucket

Purpose:

Store Terraform state.

Bucket Used:

```text
enterprise-platform-tf-state-761018849945
```

Create:

```bash
aws s3api create-bucket \
  --bucket enterprise-platform-tf-state-761018849945 \
  --region us-east-1
```

---

# Step 2 - Enable Versioning

Purpose:

Protect against accidental state corruption.

Enable:

```bash
aws s3api put-bucket-versioning \
  --bucket enterprise-platform-tf-state-761018849945 \
  --versioning-configuration Status=Enabled
```

Benefits:

* Restore previous state versions
* Recover from mistakes
* Audit infrastructure history

---

# Step 3 - Create DynamoDB Lock Table

Purpose:

Prevent multiple Terraform executions from modifying state simultaneously.

Table:

```text
terraform-locks
```

Create:

```bash
aws dynamodb create-table \
  --table-name terraform-locks \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1
```

Verification:

```bash
aws dynamodb describe-table \
  --table-name terraform-locks \
  --region us-east-1
```

Expected:

```text
TableStatus = ACTIVE
```

---

# Step 4 - Create backend.tf

Location:

```text
enterprise-platform-infra/
└── environments/
    └── dev/
        └── backend.tf
```

Contents:

```hcl
terraform {

  backend "s3" {

    bucket = "enterprise-platform-tf-state-761018849945"

    key = "dev/terraform.tfstate"

    region = "us-east-1"

    dynamodb_table = "terraform-locks"

    encrypt = true
  }
}
```

---

# Understanding the Key

The key is the path of the state file inside the S3 bucket.

Example:

```hcl
key = "dev/terraform.tfstate"
```

Terraform stores:

```text
s3://enterprise-platform-tf-state-761018849945/dev/terraform.tfstate
```

Future environments:

```text
dev/terraform.tfstate
staging/terraform.tfstate
prod/terraform.tfstate
```

This allows one bucket to hold multiple environment states.

---

# Step 5 - Migrate Existing State

From:

```text
enterprise-platform-infra/environments/dev
```

Run:

```bash
terraform init
```

Terraform detects the new backend.

Expected prompt:

```text
Do you want to copy existing state to the new backend?
```

Answer:

```text
yes
```

Terraform uploads:

```text
terraform.tfstate
```

into S3.

---

# Step 6 - Verify Migration

Check:

```bash
aws s3 ls \
s3://enterprise-platform-tf-state-761018849945/dev/
```

Expected:

```text
terraform.tfstate
```

# Step 7 - Configure Jenkins

Jenkins must access:

1. S3 backend
2. DynamoDB lock table

For project development we temporarily attached:

```text
AdministratorAccess
```

to:

```text
jenkins-ec2-role
```

Benefits:

* Fast project completion
* Avoid IAM permission roadblocks
* Simplifies EKS creation

---

# Future Production Hardening

Remove:

```text
AdministratorAccess
```

Replace with:

```text
Jenkins-DevOps-Deployment-Policy
```

containing only:

* EKS permissions
* ECR permissions
* S3 backend permissions
* DynamoDB lock permissions
* CloudWatch permissions
* IAM PassRole permissions

This follows least-privilege security principles.
