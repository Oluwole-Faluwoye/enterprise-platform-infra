## Why GitHub SSH Credentials Are Used

In enterprise environments, automation platforms such as Jenkins require a secure machine identity to interact with source control systems.

When Jenkins needs to:

* Clone repositories
* Pull application source code
* Push GitOps configuration updates
* Access private repositories

it must authenticate to GitHub.

### Why Not Use Personal Access Tokens (PATs)

Although Personal Access Tokens can be used, they are often tied to individual user accounts.

This creates operational risks:

* Tokens may expire
* Users may leave the organization
* Credentials require manual rotation
* Automation becomes dependent on a specific employee

### Why SSH Is Preferred

SSH keys provide Jenkins with a dedicated machine identity that is independent of any individual user.

Benefits include:

* Secure repository access
* No password storage
* No token expiration concerns
* Easier automation
* Widely adopted enterprise practice
* Supports both repository cloning and GitOps updates

Architecture:

Jenkins
↓
SSH Key
↓
GitHub

### Current Project Implementation

At the current stage of the project, the GitHub repositories are public, therefore Jenkins can clone repositories without authentication.

However, the platform is designed using enterprise best practices so that repositories can be transitioned to private repositories in the future without requiring pipeline redesign.

When GitOps is introduced, Jenkins will also require permission to push changes back to GitHub, making SSH authentication the preferred long-term solution.

### Future Enterprise Enhancement

In large organizations, GitHub Apps are increasingly replacing SSH keys because they provide:

* Fine-grained permissions
* Centralized management
* Auditing capabilities
* Credential rotation

For this project, SSH authentication provides the best balance between simplicity, security, portability, and real-world enterprise adoption.

# Step 1 – Generate SSH Key Pair

SSH into the Jenkins EC2 server:

```bash
ssh -i us-east-1-key.pem ec2-user@<JENKINS-PUBLIC-IP>
```

Generate a new SSH key:

```bash
ssh-keygen -t ed25519
```

When prompted:

```text
Enter file in which to save the key   or simply press enter to use the default location
```

Press:

ENTER
```

Accept the default location:

```text
/home/ec2-user/.ssh/id_ed25519

```

Press:

ENTER
```

When prompted again:

```text
Press enter again Enter 
```


This creates:

```text
~/.ssh/id_ed25519
~/.ssh/id_ed25519.pub
```

---

# Step 2 – Verify Key Creation

Run:

```bash
ls -la ~/.ssh
```

Expected output:

```text
id_ed25519
id_ed25519.pub
```

Files:

```text
id_ed25519      = Private Key
id_ed25519.pub  = Public Key

NOTE : 

Infrastructure deployments are protected by a dual-control mechanism consisting of a deployment parameter (APPLY_CHANGES) and a manual approval gate. By default, the pipeline performs planning and security validation only. Infrastructure modifications require both explicit deployment intent and human approval before execution. This reduces the risk of unintended infrastructure changes and aligns with enterprise change-management practices.
